# Hackathon Roadmap — Annotation Platform

## Summary

A self-hosted annotation platform where developers capture visual and technical observations from the browser during development (worktrees, preview environments), enriched with project-specific context via hooks generated by Claude Code. Annotations are stored as W3C Web Annotations in Postgres on a team server, pushed in real-time into active Claude Code sessions via channels, and triaged by agents — resolved directly or filed into tracking systems. The human observes, the agent acts.

## Prior Art

The core components exist as working internal tools and will be extracted/repackaged:

- **overlay.js** (1,888 lines) — Browser overlay with Fabric.js markup (rect, arrow, text, highlight), Cropper.js region capture, element picker, responsive viewports, annotation CRUD. Currently injected via Caddy on `*.dev.handgemacht.ai`. Location: `/.runtime/tailnet-ingress/overlay/overlay.js`
- **Go overlay-api** (995 lines) — HTTP server handling annotation pipeline, image storage, bead integration. Currently proxies to Spotter's Elixir API. Location: `/.runtime/tailnet-ingress/overlay/overlay-api/main.go`
- **Spotter annotation system** — Ash resource, REST API, MCP tools, image storage, review UI. Production-grade. The data model and MCP surface inform the new product but are not carried over — we start clean with W3C.

## Features

### Chrome Extension — Capture UX

**Scope**: Manifest V3 Chrome extension. Developer selects a region on any page (drag-to-select, resize handles to adjust), annotates with drawing tools (rect, arrow, text, highlight), and saves with a text comment. Captures a cropped screenshot of the selected region. Keyboard shortcut to trigger.

**Extracted from**: `overlay.js` — Fabric.js markup, Cropper.js capture, element picker carry over directly. Remove subdomain-based project/worktree detection, replace Go overlay-api calls with direct Go server calls, package as Manifest V3 (content script, background service worker, side panel).

**User groups**: Developers doing QA passes on their own work or reviewing teammate's work in worktrees/preview environments.

**Boundaries**:
- IN: Drag-to-select region capture, resize handles before snapshot, drawing tools (rect, arrow, text, highlight), text comment, keyboard shortcut to trigger, undo support
- OUT: Full-page capture, scrolling capture, video recording, session replay, interaction buffer (v2)

**Dependencies**: Go server must be running to receive and store annotations.

**Open questions**: Exact drawing tool set — what's the minimum that feels complete? The existing overlay has rect, arrow, highlight. Text tool exists but needs verification.

### Chrome Extension — Responsive Viewport Annotation

**Scope**: The extension provides viewport simulation at configurable responsive sizes (mobile, tablet, desktop, custom), allowing developers to annotate at each breakpoint. Renders the page in iframes at different sizes. Each annotation records the viewport dimensions as metadata.

**Extracted from**: `overlay.js` — existing device presets (iPhone SE 375x812, iPhone 14 390x844, iPad 768x1024, Desktop SM 1024x768) and iframe-based responsive preview carry over.

**User groups**: Developers reviewing responsive behavior across breakpoints before pushing.

**Boundaries**:
- IN: Predefined breakpoints plus custom sizes, viewport size recorded as annotation metadata, annotations visible per-viewport
- OUT: Emulating device-specific behavior (touch events, user agent), real device testing replacement

**Dependencies**: Chrome Extension — Capture UX.

**Open questions**: Single viewport with size switching vs. multiple viewports side-by-side? Performance impact of rendering multiple iframes?

### Chrome Extension — Annotation List, CRUD, and Filtering

**Scope**: The extension displays a browsable list of all annotations in a side panel. Full CRUD — create, view, edit, delete. Filter bar with quick filters. Annotations display metadata chips (mobile/desktop, worktree name, author). Annotations scoped to current domain by default, with domain switcher for viewing annotations from other domains.

**User groups**: Developers reviewing the state of annotations on a feature branch or preview environment.

**Boundaries**:
- IN: Side panel with list view, filter bar (viewport size, worktree, author, status), domain scoping with switcher, metadata chips per annotation, edit comment/drawings, delete
- OUT: Kanban/board views, priority sorting, assignment — the extension is not a project management tool

**Dependencies**: Go server API for listing, updating, deleting annotations.

### Chrome Extension — Automatic Context Capture

**Scope**: Every annotation automatically captures a baseline set of technical context alongside the screenshot and comment. Enough for Claude Code to understand the environment without the developer describing it manually.

**User groups**: Developers capturing, AI agents consuming annotations.

**Boundaries**:
- IN (v1 baseline): URL, viewport dimensions, timestamp, user agent, console errors and warnings (at moment of annotation), failed network requests (4xx/5xx at moment of annotation), core web vitals snapshot
- IN (extensibility): Designed to be extended via hook system — baseline fields are well-known and indexed, custom fields go into extras
- OUT: Continuous console/network recording, session replay, interaction buffer (v2)

**Dependencies**: None — core extension functionality.

**Open questions**: Console log capture depth — last N entries, last N seconds, or everything since page load?

### Hook System — Project-Specific Enrichment

**Scope**: The extension calls `/__annotation_context` on the current page's dev server when capturing an annotation. If the endpoint exists, the response JSON is merged into the annotation metadata. If it doesn't exist (404, timeout at 500ms), the annotation proceeds without enrichment — graceful degradation, no error shown to user.

**User groups**: Developers who want richer annotation context (worktree path, git branch, framework-specific state, app-specific data like video frame numbers or A/B test variants).

**Boundaries**:
- IN: Well-known endpoint convention, JSON response merged into annotation, known fields (worktree path, git branch, git commit, port) get indexed in the database, everything else goes into JSONB extras
- OUT: Authentication on the hook endpoint (localhost/dev only), complex negotiation or versioning of the hook contract

**Dependencies**: The Claude Code plugin's setup-hooks skill generates the middleware. The extension just calls the endpoint and merges the result.

**Open questions**: Should the endpoint return information about available custom fields so the extension's filter bar can offer them? Probably overengineering for v1.

### Go Server — Storage and API

**Scope**: Go HTTP server storing annotations in Postgres, serving them via REST API. Annotations follow the W3C Web Annotation data model natively — JSON-LD in a `jsonb` column with indexed columns for known fields. Screenshots stored in Postgres behind a storage interface. Deployed via Docker Compose.

**Extracted from**: `overlay-api/main.go` — annotation struct definitions, validation, CRUD handlers, image handling, metadata mapping carry over. Replace Spotter HTTP proxy calls with direct Postgres writes. Reshape data envelope from custom format to W3C Web Annotation.

**User groups**: Chrome extension (writes), Claude Code via MCP (reads/resolves), team administrators (deployment).

**Schema**:
```sql
CREATE TABLE annotations (
  id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  annotation    JSONB NOT NULL,       -- W3C Web Annotation JSON-LD (canonical)
  project       TEXT NOT NULL,
  domain        TEXT,
  worktree      TEXT,
  branch        TEXT,
  state         TEXT NOT NULL DEFAULT 'open',
  motivation    TEXT NOT NULL DEFAULT 'commenting',
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  modified_at   TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE annotation_images (
  annotation_id UUID PRIMARY KEY REFERENCES annotations(id) ON DELETE CASCADE,
  image_data    BYTEA NOT NULL,
  content_type  TEXT NOT NULL DEFAULT 'image/png',
  size_bytes    INTEGER NOT NULL,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

**Boundaries**:
- IN: Create annotation (with screenshot upload), list annotations (filters: domain, worktree, status, viewport, author), get by ID, get screenshot, update, delete, resolve (with generic metadata), CORS for extension, webhook to channel server on new annotation
- OUT: Full-text search (agent filters client-side), aggregation/analytics endpoints

**Dependencies**: Postgres (Docker Compose).

**Open questions**: Rate limiting? Max screenshot size? Annotation retention/cleanup policy?

### Go Server — MCP Server

**Scope**: The Go server exposes MCP tools over HTTP transport. The agent brings its own context (worktree, branch) collected by an environment script.

**Boundaries**:
- IN: `list_annotations` (filterable by domain, worktree, status, viewport, author, time range), `get_annotation_image` (screenshot for annotation ID), `resolve_annotation` (marks resolved with generic metadata — bead ID, commit hash, PR link, whatever the agent passes)
- OUT: Complex query language, semantic search, annotation creation via MCP, bulk operations

**Dependencies**: Go server API.

### Claude Code Channel Plugin

**Scope**: A Claude Code channel server (Bun) that bridges the Go server to an active Claude Code session. When a new annotation is created, the Go server webhooks the channel, and the channel pushes the event into the session in real-time. Two-way: Claude resolves annotations back through the channel.

**This is new** — not extracted from prior art. Based on Claude Code channels (research preview, v2.1.80+).

**Flow**:
```
Chrome Extension
    |  POST /api/annotations (+ image)
    v
Go Server (stores in Postgres)
    |  POST webhook to channel server
    v
Channel Server (Bun, spawned by Claude Code)
    |  mcp.notification('notifications/claude/channel', ...)
    v
Active Claude Code Session
    |  Uses MCP tools: list_annotations, get_annotation_image
    |  Reads screenshot + structured context
    v
Claude acts: fixes code, resolves annotation, or files into tracking
```

**Boundaries**:
- IN: Channel capability (`claude/channel`), webhook receiver, MCP tools, reply tool for resolving back, instructions for Claude
- OUT: Permission relay (not relevant for this use case)

**Dependencies**: Go server webhook on annotation create. Claude Code v2.1.80+.

### Claude Code Plugin — Skills and Environment

**Scope**: Claude Code plugin bundling MCP config, skills, and environment collection.

**Boundaries**:
- IN: MCP server config pointing at Go server (HTTP transport, does not start the server), setup-hooks skill (detects framework — Next.js, Vite, Phoenix — asks what context to expose, generates `/__annotation_context` middleware), environment collection bash script (worktree path, git branch, git commit, repo name — passed to MCP queries as filters), installable via `/plugin install`
- OUT: Managing Go server lifecycle, auto-updating server, extension installation

**Dependencies**: Go server running and accessible. Chrome extension installed separately.

### W3C Web Annotation Data Model

**Scope**: Native from day one. Pragmatic adoption — W3C structure with `@context`, not full linked data ceremony.

**Mapping from overlay capture data**:

| Capture data | W3C representation |
|---|---|
| Comment text | `body[].TextualBody` with `purpose: "commenting"` |
| Screenshot | `body[].Image` (URL reference to server endpoint) |
| Page URL | `target.source` |
| CSS selector | `target.selector[].CssSelector` |
| Element rect | `target.selector[].FragmentSelector` (xywh= media fragment) |
| Drawn markup (arrows, boxes) | `target.selector[].SvgSelector` |
| Viewport dimensions | Recorded in target state |
| Console errors, network failures | `body[]` with `purpose: "describing"` |
| Hook-provided data | Extension namespace for project-specific fields |

**Boundaries**:
- IN: Body, target, selectors (CSS, Fragment, SVG), motivation, creator, timestamps, JSON-LD serialization with `@context`
- OUT: Full W3C compliance on every optional field — use what maps to the tool's needs

**Open questions**: How closely to follow JSON-LD serialization vs. simplified JSON that's W3C-compatible but easier to work with?

### Docker Compose Deployment

**Scope**: `docker-compose.yml` bringing up Go server + Postgres with `docker compose up`. Sensible defaults, env var config, persistent volume mounts.

**Boundaries**:
- IN: Localhost demo mode (no auth, pre-configured for localhost, clone → `docker compose up` → install extension → annotating in 5 minutes)
- IN: Environment variable slots for Google SSO credentials (when implemented)
- OUT: Production deployment guides, managed hosting, Kubernetes

## Not in Weekend Scope

- **Google SSO** — design for it (auth interface in Go), don't implement
- **Pattern detection / analytics** — annotation stream analysis over time
- **Interaction buffer / session replay** — continuous recording (v2)
- **DevTools MCP for agent-initiated capture** — agent drives Chrome to create annotations (v2)
- **Issue tracker integrations** — Jira, Linear, GitHub Issues sync
- **Multi-provider OAuth, RBAC** — Google only, flat permissions
- **Browsers beyond Chrome**
- **Mobile**
- **Performance optimization** — premature before users
