#### 0.2 — CLAUDE.md files

Each subdirectory gets a CLAUDE.md so any session landing there has full context.

**`annotation-plugin/CLAUDE.md`** (root):
- Project purpose (one paragraph from ROADMAP.md summary)
- How to run: `just up` starts Docker Compose (Postgres), `just server` runs Go in dev mode
- Port assignments from `.worktree-ports.json`
- Link to API.md for the contract
- Link to ROADMAP.md for full scope
- Test commands: `just test` (all), `just test-server`, `just test-ext`
- "Read API.md before writing any endpoint or API call"

**`server/CLAUDE.md`**:
- Go conventions: standard library HTTP (no framework), `internal/` for non-exported packages
- Test convention: use scenarigo for integration tests
- DB: Postgres via `pgx`, no ORM
- Migrations: plain SQL files in `migrations/`, applied in order
- W3C Web Annotation: the `annotation` JSONB column is the canonical W3C envelope; indexed columns (`project`, `domain`, `worktree`, `branch`, `state`, `motivation`) are denormalized for queries

**`extension/CLAUDE.md`**:
- Manifest V3 patterns: content script for page injection, background service worker for lifecycle, side panel for annotation list
- No build step for v1 — plain JS, loaded unpacked via `chrome://extensions`
- Fabric.js for canvas markup, Cropper.js for region selection
- API base URL configured in background service worker, defaults to `http://localhost:PORT`
- Content script communicates with background via `chrome.runtime.sendMessage`

#### 0.3 — Claude rules (`.claude/rules/`)

**`w3c-annotations.md`** — W3C vocabulary reference:
- Annotation envelope structure (`@context`, `id`, `type`, `body`, `target`, `creator`, `created`, `modified`, `motivation`)
- Body types used: `TextualBody` (comments), `Image` (screenshot URL), describing bodies (console errors, network failures)
- Target structure: `source` (URL), `selector` array (CssSelector, FragmentSelector with xywh=, SvgSelector for drawings)
- Motivation values: `commenting` (default), `highlighting`, `describing`
- "Always store the full W3C envelope in the `annotation` JSONB column. Never flatten W3C fields into top-level SQL columns."

**`api-conventions.md`** — REST conventions:
- Resource naming: `/api/annotations`, `/api/annotations/:id`, `/api/annotations/:id/image`
- JSON envelope: `{ "data": ... }` for single, `{ "data": [...], "meta": { "count": N } }` for lists
- Errors: `{ "error": { "code": "not_found", "message": "..." } }`
- Timestamps: RFC 3339
- IDs: UUIDs

**`testing.md`** — Test conventions:
- Go server: scenarigo for HTTP integration tests against real Postgres
- Extension: manual testing via Chrome unpacked extension (no unit test framework for v1)
- Integration: end-to-end test script that creates, lists, gets, deletes an annotation

#### 0.4 — Docker Compose skeleton

## Epic 1: API Contract & Schema Design

**Owner**: Session A
**Phase**: 0 (starts immediately, parallel with Epic 0)
**Duration**: 30 minutes
**Output**: `API.md` committed and pushed — the single source of truth for both server and extension

### Tasks

#### 1.1 — Write API.md

Document every endpoint with method, path, request body, response body, and status codes.

**Endpoints to define**:

| Method | Path | Purpose |
|--------|------|---------|
| `POST` | `/api/annotations` | Create annotation (multipart: JSON metadata + image file) |
| `GET` | `/api/annotations` | List annotations (query params: domain, worktree, branch, state, motivation, viewport, creator, limit, offset) |
| `GET` | `/api/annotations/:id` | Get single annotation |
| `GET` | `/api/annotations/:id/image` | Get screenshot image (returns image/png) |
| `PUT` | `/api/annotations/:id` | Update annotation (comment, state, metadata) |
| `DELETE` | `/api/annotations/:id` | Delete annotation |
| `POST` | `/api/annotations/:id/resolve` | Resolve annotation (accepts generic metadata: commit, PR, bead ID) |

#### 1.2 — Define W3C annotation envelope

Document the exact JSON-LD shape for create requests and stored annotations. Include concrete examples for:
- Simple text comment with screenshot
- Annotation with drawing markup (SvgSelector)
- Annotation with hook-provided context (extras)

#### 1.3 — Define Postgres schema

Finalize the schema from the roadmap. Include:
- `annotations` table with indexed columns
- `annotation_images` table
- Index definitions for common query patterns
- Migration SQL file

#### 1.4 — Define error codes

Enumerate all error responses: validation errors, not found, conflict, server error.

### Completion criteria

- [ ] API.md covers all endpoints with request/response examples
- [ ] W3C envelope documented with 3+ concrete examples
- [ ] Schema SQL finalized and included
- [ ] Committed and pushed

## Epic 2: Go Server Foundation

**Owner**: Session A
**Phase**: 1 (starts after Epic 1 is committed)
**Duration**: 3–4 hours
**Dependencies**: Epic 0 (repo structure), Epic 1 (API contract)
**Prior art**: `/.runtime/tailnet-ingress/overlay/overlay-api/main.go` (995 lines)

### Tasks

#### 2.1 — Database layer

- Migration files: create `annotations` and `annotation_images` tables
- Connection pool setup with `pgx`
- Migration runner (apply SQL files in order on startup)

#### 2.2 — Storage interface

```go
type Store interface {
    CreateAnnotation(ctx, annotation) (Annotation, error)
    ListAnnotations(ctx, filters) ([]Annotation, int, error)
    GetAnnotation(ctx, id) (Annotation, error)
    UpdateAnnotation(ctx, id, update) (Annotation, error)
    DeleteAnnotation(ctx, id) error
    ResolveAnnotation(ctx, id, resolution) (Annotation, error)
    SaveImage(ctx, annotationID, data, contentType) error
    GetImage(ctx, annotationID) ([]byte, string, error)
}
```

Postgres implementation. No interface gymnastics — one implementation, but the boundary is clean.

#### 2.3 — HTTP handlers

Implement all endpoints from API.md:
- `POST /api/annotations` — multipart form: `annotation` (JSON) + `image` (file)
- `GET /api/annotations` — query param filtering, pagination
- `GET /api/annotations/:id` — single fetch
- `GET /api/annotations/:id/image` — serve image bytes with content-type
- `PUT /api/annotations/:id` — partial update
- `DELETE /api/annotations/:id` — hard delete
- `POST /api/annotations/:id/resolve` — set state to resolved, store resolution metadata

#### 2.4 — W3C annotation model

- Go structs matching the W3C envelope from API.md
- JSON marshaling/unmarshaling
- Validation: required fields (`body`, `target.source`), valid motivation values
- Auto-populated fields: `id`, `created`, `modified`, `creator` (from header or default)

#### 2.5 — CORS middleware

- Allow origins: `chrome-extension://*`, `http://localhost:*`, configurable via env
- Allow methods: GET, POST, PUT, DELETE, OPTIONS
- Allow headers: Content-Type, Authorization
- Preflight caching

#### 2.6 — Server entrypoint

- `cmd/server/main.go`: config from env vars, DB connection, migration, router, graceful shutdown
- Health endpoint: `GET /health`
- Fail fast on missing config

#### 2.7 — Webhook stub

- On annotation create, POST to configurable webhook URL (env var)
- Fire-and-forget, log errors, don't block the response
- This enables the channel server on Sunday without touching server code

### Completion criteria

- [ ] `just up && just db-migrate && just server` starts cleanly
- [ ] All CRUD endpoints work via curl
- [ ] Image upload and retrieval works
- [ ] CORS headers present on all responses
- [ ] Webhook fires on create (to a test endpoint or logs)

### Reference material

Read `/.runtime/tailnet-ingress/overlay/overlay-api/main.go` for:
- Annotation struct patterns (adapt to W3C)
- Image handling (multipart parsing, storage)
- HTTP handler patterns
- Do NOT copy Spotter proxy calls — replace with direct Postgres

## Epic 3: Chrome Extension — Capture Core

**Owner**: Session B
**Phase**: 1 (starts after pulling Epic 0 scaffold)
**Duration**: 3–4 hours
**Dependencies**: Epic 0 (repo structure, extension CLAUDE.md)
**Prior art**: `/.runtime/tailnet-ingress/overlay/overlay.js` (1,888 lines)

### Tasks

#### 3.1 — Manifest V3 scaffold

```json
{
  "manifest_version": 3,
  "name": "Annotation Platform",
  "version": "0.1.0",
  "permissions": ["activeTab", "sidePanel", "storage"],
  "content_scripts": [{
    "matches": ["<all_urls>"],
    "js": ["src/content/content.js"],
    "css": ["src/content/content.css"],
    "run_at": "document_idle"
  }],
  "background": {
    "service_worker": "src/background/background.js"
  },
  "side_panel": {
    "default_path": "src/sidepanel/sidepanel.html"
  },
  "commands": {
    "toggle-capture": {
      "suggested_key": { "default": "Ctrl+Shift+A", "mac": "Command+Shift+A" },
      "description": "Start annotation capture"
    }
  }
}
```

#### 3.2 — Content script: capture activation

- Keyboard shortcut triggers capture mode
- Dim overlay covers the page
- Drag-to-select draws a region rectangle
- Resize handles on the selection rectangle
- ESC cancels capture mode
- Extract relevant logic from `overlay.js` lines handling Cropper.js initialization and region selection

#### 3.3 — Content script: markup tools

- After region is selected, show toolbar above/below selection
- Tools: rectangle, arrow, highlight (semi-transparent rect), text
- Fabric.js canvas overlay positioned over the selected region
- Each tool: click-drag to draw, color picker (red default), undo (Ctrl+Z)
- Extract from `overlay.js` Fabric.js tool implementations

#### 3.4 — Content script: screenshot capture

- After markup is complete, user clicks "Save" or hits Enter
- Use Cropper.js or canvas API to capture the selected region as PNG
- Composite: page screenshot (via background service worker `chrome.tabs.captureVisibleTab`) cropped to selection, with Fabric.js markup overlaid
- Send image blob + metadata to background service worker

#### 3.5 — Content script: comment input

- After capture, show a text input overlay for the annotation comment
- Submit with Enter (Shift+Enter for newline) or Save button
- Cancel with ESC
- Package: comment text + screenshot blob + page URL + viewport dimensions + timestamp

#### 3.6 — Background service worker

- Listen for messages from content script
- `captureVisibleTab` for screenshot base
- Store server URL in `chrome.storage.sync` (default `http://localhost:8090`)
- Forward create-annotation requests to server (Phase 2 wiring, stub for now)
- Side panel management

#### 3.7 — Side panel shell

- Basic HTML/CSS for the annotation list (empty state for now)
- Will be populated in Epic 4 / Phase 2
- Settings: server URL configuration

### Completion criteria

- [ ] Extension loads in Chrome via `chrome://extensions` (unpacked)

- [ ] Drawing tools render on canvas (rect, arrow, highlight, text)
